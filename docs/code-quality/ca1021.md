# <a name="ca1021-avoid-out-parameters"></a>CA1021: evitar parâmetros de saída

|||
|-|-|
|TypeName|AvoidOutParameters|
|CheckId|CA1021|
|Categoria|Microsoft.Design|
|Mudança de ruptura|Quebra|

## <a name="cause"></a>Causa
Um método público ou protegido em `out` um tipo público tem um parâmetro.

Por padrão, esta regra só olha para tipos visíveis externamente, mas isso é [configurável](#configurability).

## <a name="rule-description"></a>Descrição da regra
Passar tipos por `out` referência `ref`(usando ou ) requer experiência com ponteiros, entender como os tipos de valor e os tipos de referência diferem e métodos de manuseio com múltiplos valores de retorno. Além disso, `out` a `ref` diferença entre e parâmetros não é amplamente compreendida.

Quando um tipo de referência é aprovado "por referência", o método pretende usar o parâmetro para retornar uma instância diferente do objeto. Passar um tipo de referência por referência também é conhecido como usar um ponteiro duplo, ponteiro para um ponteiro ou dupla indireção. Usando a convenção de chamada padrão, que é passar "por valor", um parâmetro que leva um tipo de referência já recebe um ponteiro para o objeto. O ponteiro, não o objeto para o qual ele aponta, é passado por valor. Passar por valor significa que o método não pode alterar o ponteiro para que ele aponte para uma nova instância do tipo de referência. No entanto, ele pode alterar o conteúdo do objeto para o qual ele aponta. Para a maioria das aplicações isso é suficiente e produz o comportamento desejado.

Se um método deve retornar uma instância diferente, use o valor de retorno do método para conseguir isso. Consulte <xref:System.String?displayProperty=fullName> a classe para uma variedade de métodos que operam em strings e retorne uma nova instância de uma string. Quando este modelo é usado, o chamador deve decidir se o objeto original está preservado.

Embora os valores de retorno sejam comuns `out` `ref` e muito utilizados, a aplicação correta e os parâmetros exigem habilidades intermediárias de design e codificação. Arquitetos de bibliotecas que projetam para um `out` público `ref` geral não devem esperar que os usuários dominem o trabalho com ou parâmetros.

## <a name="how-to-fix-violations"></a>Como corrigir violações
Para corrigir uma violação desta regra que é causada por um tipo de valor, tenha o método para retornar o objeto como seu valor de retorno. Se o método deve retornar vários valores, redesenhe-o para retornar uma única instância de um objeto que detém os valores.

Para corrigir uma violação desta regra causada por um tipo de referência, certifique-se de que o comportamento desejado é retornar uma nova instância da referência. Se for, o método deve usar seu valor de retorno para fazer isso.

## <a name="when-to-suppress-warnings"></a>Quando suprimir avisos
É seguro suprimir um aviso desta regra. No entanto, esse projeto pode causar problemas de usabilidade.

## <a name="configurability"></a>Configurabilidade

Se você estiver executando esta regra a partir de [analisadores FxCop](install-fxcop-analyzers.md) (e não com análise de legado), você pode configurar quais partes da sua base de código para executar essa regra, com base em sua acessibilidade. Por exemplo, para especificar que a regra deve ser executada apenas contra a superfície da API não pública, adicione o seguinte par de valorde chave a um arquivo .editorconfig em seu projeto:

```ini
dotnet_code_quality.ca1021.api_surface = private, internal
```

Você pode configurar esta opção apenas para esta regra, para todas as regras ou para todas as regras nesta categoria (Design). Para obter mais informações, consulte [Configure analisadores FxCop](configure-fxcop-analyzers.md).
