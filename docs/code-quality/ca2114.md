# <a name="ca2114-method-security-should-be-a-superset-of-type"></a>CA2114: a segurança de método deve ser um superconjunto de tipo

|||
|-|-|
|NomeDoTipo|MethodSecurityShouldBeASupersetOfType|
|CheckId|CA2114|
|Categoria|Microsoft.Security|
|Alteração significativa|Em interrupção|

## <a name="cause"></a>Causa
Um tipo tem segurança declarativa e um de seus métodos tem segurança declarativa para a mesma ação de segurança, e a ação de segurança não é uma [demanda de vínculo](/dotnet/framework/misc/link-demands)e as permissões verificadas pelo tipo não são um subconjunto das permissões verificadas pelo método.

> [!NOTE]
> Esta regra herdada não está disponível nos analisadores do FxCop. Para obter mais informações, consulte [regras preteridas](fxcop-rule-port-status.md#deprecated-rules).

## <a name="rule-description"></a>Descrição da regra
Um método não deve ter uma segurança declarativa em nível de método e de nível de tipo para a mesma ação. As duas verificações não são combinadas; somente a demanda de nível de método é aplicada. Por exemplo, se um tipo exigir permissão `X`e um de seus métodos exigir permissão `Y`, o código não precisará ter a permissão `X` para executar o método.

## <a name="how-to-fix-violations"></a>Como corrigir violações
Examine seu código para certificar-se de que ambas as ações são necessárias. Se ambas as ações forem necessárias, verifique se a ação no nível do método inclui a segurança especificada no nível de tipo. Por exemplo, se o tipo exige permissão `X`e seu método também deve solicitar a permissão `Y`, o método deve solicitar explicitamente `X` e `Y`.

## <a name="when-to-suppress-warnings"></a>Quando suprimir avisos
É seguro suprimir um aviso dessa regra se o método não exigir a segurança especificada pelo tipo. No entanto, esse não é um cenário comum e pode indicar uma necessidade de uma revisão cuidadosa do design.
